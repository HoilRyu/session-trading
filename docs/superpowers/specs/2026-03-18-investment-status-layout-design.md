# 투자현황 탭 영역 구분 설계서

## 개요

- 작성일: 2026-03-18
- 대상 경로:
  - `frontend/src/pages/InvestmentStatusPage.tsx`
  - `frontend/src/features/investment-status/`
  - 필요 시 `frontend/src/app/`, `frontend/src/test/`
- 목적: 현재 플레이스홀더 상태인 `투자 현황` 탭을 실제 레이아웃 구조로 전환하기 전에, 데스크톱 기준의 큰 영역 경계와 컴포넌트 책임을 먼저 확정한다.

## 목표

- `투자 현황` 탭을 `상단 작업현황 레이어 + 하단 투자방식 보드 + 우측 상세 패널` 구조로 정의한다.
- 상단 작업현황은 본문과 시각적으로 분리된 독립 레이어로 보여야 한다.
- 상단 작업현황 내부의 각 작업은 `가로로 긴 리스트형 카드`로 표현해야 한다.
- 하단 좌측에는 코드로 정의된 투자방식 카드만 노출하고, 카드 클릭 시 우측 상세 패널이 열린다.
- 이번 단계에서는 카드 내부의 상세 수치 설계보다 `영역 구분`, `레이아웃 비율`, `컴포넌트 경계`를 우선 확정한다.

## 범위

- 데스크톱 `투자 현황` 탭 레이아웃 구조 정의
- 상단 작업현황 영역의 시각 계층 정의
- 하단 투자방식 보드와 우측 상세 패널의 영역 비율 정의
- 초기 컴포넌트 분리 방향 정의
- 클릭 흐름과 빈 상태 표현 정의
- 관련 프론트 테스트 범위 정의

## 비범위

## 사용자 확정 사항

- 데스크톱 우선으로 영역 구분을 진행한다.
- 하단 투자방식 카드는 초기 기준 `3개 이하`를 상정한다.
- 상단 `현재 작업 목록`은 `작업별 작은 카드가 나란히 있는 형태`보다, 시각적으로 더 분리된 레이어 안에서 `가로로 긴 리스트형 카드`가 쌓이는 방향이 적합하다.
- 하단 투자방식 카드는 클릭 가능한 진입 카드여야 하고, 클릭 시 상세는 `우측 패널`에서 열린다.
- UI에서 투자방식을 추가하지 않는다. 투자방식은 코드로 정의된 목록만 노출한다.
- 이번 단계에서는 투자방식 카드 내부 요약 지표보다 영역 경계 확정에 집중한다.

## 검토한 방향과 최종 선택

초기 비교안에서는 `상단 집중형`, `균형형`, `보드형`을 비교했다. 이후 사용자 피드백을 반영해 `목록 혼합형(C)`을 기반으로 아래 두 가지를 강화한 수정안으로 수렴했다.

- 작업현황을 본문과 더 분리된 별도 레이어로 강화
- 작업 카드를 세로 적층형이 아닌 `가로형 리스트 카드`로 변경

최종 방향은 `분리된 작업현황 레이어 + 하단 투자방식 보드 + 우측 상세 패널`이다.

## 설계 결정

### 1. 공용 플레이스홀더를 제거하고 투자현황 전용 레이아웃으로 분리한다

현재 `InvestmentStatusPage`는 공용 플레이스홀더를 렌더링한다. 이번 설계는 공용 `DesktopRouteContent`를 계속 확장하지 않는다.

대신 아래 구조를 권장한다.

```text
InvestmentStatusPage
└─ InvestmentStatusDesktopLayout
   ├─ RunningJobsSection
   └─ InvestmentStatusContentGrid
      ├─ StrategyBoardSection
      └─ StrategyDetailPanel
```

이렇게 분리하면:

- `투자 현황` 탭만의 시각 구조를 독립적으로 발전시킬 수 있고
- 이후 작업 데이터, 전략 요약 데이터, 상세 패널 상태를 별도 책임으로 나눌 수 있으며
- 다른 탭 플레이스홀더와 결합되지 않는다.

### 2. 상단 작업현황은 본문과 다른 레이어로 강하게 분리한다

상단 작업현황은 단순한 헤더가 아니라 운영 중인 작업을 즉시 인지시키는 독립 섹션이다.

핵심 규칙:

- 상단 전체 폭 사용
- 하단 본문과 다른 배경 톤 사용
- 더 강한 그림자 또는 대비로 레이어 분리
- 내부 여백을 넉넉히 잡아 본문 카드와 다른 덩어리로 보이게 구성

이 섹션은 `현재 어떤 작업이 돌고 있는가`를 보여주는 운영 패널 성격이므로, 하단 전략 카드 보드보다 우선도가 높다.

### 3. 작업현황 카드는 가로로 긴 리스트형 카드로 표현한다

작업 카드는 상단 레이어 안에서 세로로 쌓이는 `가로형 리스트 카드`로 정의한다.

각 카드가 갖는 역할:

- 한 개의 실행 작업 표현
- 이름, 상태, 진행 정보 같은 핵심만 빠르게 스캔 가능해야 함

레이아웃 규칙:

- 데스크톱에서 카드 하나는 가로로 긴 1행 카드
- 카드들은 상단 섹션 안에서 세로로 적층
- 초기 화면에서는 2~3개 작업이 한눈에 보이도록 높이 조절

이 구조가 필요한 이유:

- 작업 종류와 상태를 빠르게 비교하기 쉽고
- 세부 수치 설계가 늦어져도 영역 골격이 무너지지 않으며
- 추후 작업 수가 늘어도 리스트 패턴으로 자연스럽게 확장된다.

### 4. 하단은 좌측 전략 보드, 우측 상세 패널의 2열 구조로 간다

하단 메인 영역은 전략 선택과 상세 확인을 동시에 지원해야 한다. 따라서 단일 컬럼이 아니라 `보드 + 패널` 2열 구조를 채택한다.

권장 구조:

```text
InvestmentStatusContentGrid
├─ StrategyBoardSection
└─ StrategyDetailPanel
```

권장 비율:

- 좌측 전략 보드: 약 `62~68%`
- 우측 상세 패널: 약 `32~38%`

구현 시점에서는 `grid-cols-[minmax(0,1.35fr)_minmax(18rem,0.72fr)]` 수준의 비율이 출발점으로 적절하다.

이 비율을 권장하는 이유:

- 초기 전략 카드 수가 `3개 이하`이므로 좌측 보드가 충분한 존재감을 가져야 하고
- 우측 패널도 선택 후 정보를 보여줄 독립 공간이 필요하기 때문이다.

### 5. 좌측 전략 보드는 코드로 정의된 투자방식 카드만 보여준다

좌측 보드는 투자방식 진입 카드의 집합이다.

규칙:

- 노출되는 카드 목록은 코드에서 정의된 투자방식만 사용
- UI 안에 `새 투자방식 추가` 액션을 넣지 않음
- 카드 수가 `3개 이하`인 초기 조건을 기준으로 데스크톱에서는 3열 보드를 우선 적용
- 전략 카드가 `1개` 또는 `2개`여도 데스크톱 기본 리듬은 3열 기준을 유지하고, 카드 폭이 과도하게 늘어나지 않도록 같은 열 구조 안에서 배치한다

초기 영역 정의에서는 카드 내부 KPI를 확정하지 않는다. 대신 아래만 보장한다.

- 카드가 클릭 가능한 진입점이어야 함
- 카드가 서로 같은 크기와 계층을 가져야 함
- 선택 전/선택 후 상태를 시각적으로 구분할 수 있어야 함

### 6. 전략 카드는 요약도 보이지만, 현재 단계에서는 요약 스키마를 고정하지 않는다

사용자 의도는 `클릭해서 들어가는 진입 카드 + 요약이 일부 보이는 구조`다. 그러나 현재 작업의 핵심은 영역 확정이므로 카드 내부 정보 항목은 추후로 미룬다.

이번 설계에서 고정하는 것은 아래까지다.

- 카드 상단에 전략 이름 영역 필요
- 카드 중간에 요약 정보를 위한 본문 슬롯 필요
- 카드 하단에 상태 또는 짧은 설명 슬롯 필요

이번 설계에서 고정하지 않는 것은 아래다.

- 수익률, 세션 수, 성공률, 비중 등 어떤 KPI를 넣을지
- 수치 포맷과 라벨
- 색상 기준

즉, `요약 슬롯은 있다`까지만 정하고, `무엇을 채울지`는 다음 설계 단계로 넘긴다.

### 7. 전략 카드 클릭 시 우측 상세 패널이 열린다

전략 카드는 단순 정보 카드가 아니라 우측 상세 패널을 여는 진입점이다.

동작 규칙:

- 카드 클릭 전
  - 우측 패널은 빈 상태 또는 안내 상태 표시
- 카드 클릭 후
  - 해당 전략이 선택 상태가 됨
  - 우측 패널에 해당 전략 상세가 표시됨
- 다른 카드 클릭 시
  - 우측 패널 내용 교체
  - 선택 상태도 새 카드로 이동

표현 규칙:

- 패널은 하단 2열 안의 우측 영역에서 고정적으로 존재
- 빈 상태에서도 우측 패널은 폭을 유지해 전체 페이지 구조가 흔들리지 않게 한다
- 선택 시 내용이 채워지고, 필요하면 오른쪽에서 살짝 등장하는 전환 효과를 추가할 수 있음
- 이번 단계에서는 패널 내부 상세 항목보다 `패널이 열린다`는 구조만 확정
- 좁은 데스크톱 폭에서는 우측 패널 최소 폭을 유지하되, 전체 레이아웃이 깨질 경우 좌측 보드보다 먼저 내부 여백과 패널 콘텐츠 밀도를 줄이는 방향을 우선 적용한다

### 8. 빈 상태와 데이터 없음 상태를 미리 정의한다

영역 설계 단계에서도 빈 상태는 정의돼야 한다. 그렇지 않으면 구현 단계에서 레이아웃이 흔들린다.

#### 작업현황 빈 상태

- 현재 실행 중 작업이 없을 때는 상단 레이어를 숨기지 않는다
- 대신 `현재 실행 중인 작업이 없습니다` 같은 안내 문구와 비어 있는 패널 상태를 보여준다

이유:

- 작업현황 영역이 페이지 구조의 일부이기 때문에 유무에 따라 레이아웃이 점프하면 안 된다

#### 전략 상세 빈 상태

- 아무 카드도 선택되지 않았을 때 우측 패널은 비어 있지 않아야 한다
- `전략 카드를 선택하면 상세가 표시됩니다` 같은 안내 상태 유지

이유:

- 우측 패널의 존재를 사용자가 이해할 수 있어야 하고
- 선택 액션의 결과 위치를 미리 학습시킬 수 있다

### 9. 모바일은 이번 설계 범위에서 제외하되 축소 방향은 유지한다

이번 설계는 데스크톱 기준이다. 다만 향후 모바일 대응을 고려해, 각 섹션은 세로 적층으로 축소 가능한 구조를 유지한다.

즉:

- `작업현황 레이어`
- `전략 보드`
- `상세 패널`

이 세 단위는 각각 독립 섹션으로 이해될 수 있어야 하고, 나중에 모바일에서 세로 적층으로 재배치할 수 있어야 한다.

## 데이터 흐름

이번 단계의 구현은 실제 API 연결보다 레이아웃 검증이 우선이므로, 초기 데이터 흐름은 단순해야 한다.

```text
mock strategy list / mock running jobs
        ↓
InvestmentStatusDesktopLayout
        ├─ RunningJobsSection
        ├─ StrategyBoardSection
        └─ StrategyDetailPanel
                ↑
        selectedStrategyId
```

동작 순서:

1. 레이아웃 컴포넌트가 목업 작업 목록과 전략 목록을 받음
2. 상단 작업현황은 그대로 렌더링
3. 좌측 전략 카드 클릭 시 `selectedStrategyId` 갱신
4. 우측 상세 패널은 선택된 전략 기준으로 렌더링

이 흐름은 이후 실제 API나 상태 저장소로 바뀌더라도 상위 조합 책임을 유지하기 쉽다.

목업 데이터 최소 필드:

- 작업 카드
  - `id`
  - `label`
  - `status`
  - `progressText`
- 전략 카드
  - `id`
  - `label`
  - `summary`
  - `description`

## 인터랙션 규칙

- 작업현황 카드는 이번 단계에서 정보 표시 중심으로 둔다
- 전략 카드는 마우스 hover와 선택 상태를 가져야 한다
- 전략 카드를 클릭하면 우측 패널 내용이 바뀐다
- 우측 패널은 페이지 바깥 모달이 아니라 하단 레이아웃 안의 고정 영역으로 유지한다

## 오류 처리와 안전 장치

이번 작업은 레이아웃 중심이지만, 구현 전제로 아래를 정의한다.

- 작업 목록이 비어 있어도 상단 레이어는 유지
- 전략 목록이 비어 있으면 좌측 보드에 `정의된 투자방식이 없습니다` 상태를 보여줌
- 선택된 전략이 목록에서 사라지면 선택 상태를 초기화하고 우측 패널을 빈 상태로 전환

이 세 규칙을 먼저 두면 구현 시점의 예외 처리가 단순해진다.

## 테스트 전략

최소 테스트 범위:

- `InvestmentStatusPage`가 공용 플레이스홀더 대신 전용 데스크톱 레이아웃을 렌더링하는지
- 작업현황 영역, 전략 보드, 우측 상세 패널 3개 주요 영역이 모두 렌더링되는지
- 작업 카드가 가로형 리스트 구조로 렌더링되는지 확인 가능한 DOM 마커가 있는지
- 전략 카드 클릭 시 우측 상세 패널 내용 또는 선택 상태가 바뀌는지
- 작업/전략 빈 상태가 각각 의도한 안내 문구를 보이는지

권장 테스트 파일:

- `frontend/src/features/investment-status/components/InvestmentStatusDesktopLayout.test.tsx`
- 필요 시 `frontend/src/pages/InvestmentStatusPage.test.tsx`

## 구현 전환 기준

이 설계가 승인되면 다음 구현 계획은 아래 순서로 가는 것이 적절하다.

1. `InvestmentStatusPage`를 전용 레이아웃 컴포넌트로 연결
2. 상단 작업현황 레이어와 가로형 작업 카드 구현
3. 하단 전략 카드 보드 구현
4. 우측 상세 패널과 선택 상태 연결
5. 목업 데이터 기반 테스트 추가

이 순서는 `영역 골격 -> 선택 상호작용 -> 테스트` 흐름을 유지해 레이아웃 작업을 안전하게 진행하게 해준다.
