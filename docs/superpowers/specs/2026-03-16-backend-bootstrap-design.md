# 백엔드 부트스트랩 설계서

## 개요

- 작성일: 2026-03-16
- 대상 경로: `backend/`
- 목적: `backend` 폴더에 `FastAPI` 기반 최소 실행 골격을 구성하기

## 목표

- 백엔드 개발을 시작할 수 있는 최소 실행 환경을 만든다.
- 프런트엔드와 연결 가능한 HTTP API 서버 시작점을 만든다.
- 이후 거래소 연동, 전략 판단, 주문 실행 기능을 무리 없이 확장할 수 있는 구조를 만든다.

## 범위

- `backend`를 `FastAPI` 기반 Python 프로젝트로 초기화하기
- 앱 시작점과 라우터 등록 구조 만들기
- 최소 헬스체크 API 만들기
- 환경설정 로딩 구조 만들기
- 기본 테스트 환경과 스모크 테스트 추가하기
- 로컬 실행 방법을 문서화하기

## 비범위

- 거래소 REST/WebSocket 연동
- 전략 엔진 구현
- 주문 실행 로직
- 데이터베이스 연결
- 인증/인가
- Docker 및 배포 설정

## 설계 결정

### 1. 프레임워크 선택

- 백엔드는 `FastAPI` 단일 서버로 구성한다.
- 이 프로젝트는 실시간 시세 처리, 전략 계산, 주문 조건 판단 등 Python 생태계의 장점을 활용할 가능성이 높다.
- 프런트엔드와의 타입 통일보다는 실시간 데이터 처리와 전략 확장성이 더 중요하므로 `NestJS` 대신 `FastAPI`를 채택한다.

### 2. 프로젝트 구조

아래처럼 최소 책임 분리 구조를 사용한다.

```text
backend/
  app/
    __init__.py
    main.py
    api/
      __init__.py
      router.py
      routes/
        __init__.py
        health.py
    core/
      __init__.py
      config.py
  tests/
    test_health.py
  pyproject.toml
  .env.example
  README.md
```

- `app/main.py`: `FastAPI` 앱 생성과 전체 라우터 등록만 담당한다.
- `app/api/router.py`: 개별 API 라우터를 묶는 루트 라우터만 담당한다.
- `app/api/routes/health.py`: 서버 생존 여부를 확인하는 최소 엔드포인트만 담당한다.
- `app/core/config.py`: 앱 이름, 환경, 기본 URL prefix 같은 설정 로딩만 담당한다.
- `tests/test_health.py`: 헬스체크 엔드포인트 스모크 테스트만 담당한다.

이 구조는 지금 단계에서 과한 추상화를 피하면서도, 나중에 `services`, `clients`, `websocket`, `strategies` 같은 디렉터리를 자연스럽게 추가할 수 있도록 한다.

### 3. 의존성 선택

- 패키지 관리는 `pyproject.toml` 기반으로 정의한다.
- 런타임은 `fastapi`, `uvicorn`, `pydantic-settings`를 사용한다.
- 테스트는 `pytest`와 `httpx`를 사용한다.

이번 단계에서는 포맷터, 린터, ORM, 마이그레이션 도구는 도입하지 않는다. 최소 골격의 목적은 실행 가능한 서버 시작점과 테스트 가능성 확보에 있다.

### 4. API 구조

- 초기 엔드포인트는 `GET /health` 하나만 둔다.
- 응답은 문자열 대신 구조화된 JSON으로 반환한다.
- 최소 응답 필드는 아래 수준으로 제한한다.
  - `status`
  - `service`
  - `environment`

이 응답은 프런트엔드 연결 전 서버 상태를 빠르게 확인하는 용도이며, 향후 버전 정보나 의존성 상태를 추가할 수 있다.

### 5. 설정 방식

- 환경설정은 `.env` 파일과 시스템 환경변수에서 읽는다.
- 설정 모델은 `pydantic-settings` 기반 클래스로 정의한다.
- 최소 설정 항목은 아래만 포함한다.
  - `app_name`
  - `environment`
  - `api_v1_prefix`

지금 단계에서는 거래소 API 키, 시크릿, 주문 옵션 같은 민감 설정을 정의하지 않는다. 그런 값은 실제 연동 시점에 별도 설계로 추가한다.

### 6. 테스트 방식

- 테스트는 `pytest` 기반으로 구성한다.
- `FastAPI` 앱에 대해 헬스체크 엔드포인트가 정상 응답하는지만 검증한다.
- 검증 범위는 다음으로 제한한다.
  - HTTP 상태 코드가 200인지 확인
  - 응답 JSON에 최소 필드가 존재하는지 확인

이 테스트는 기능 완성도 검증이 아니라 백엔드 골격이 깨지지 않았는지 확인하는 스모크 테스트 역할을 한다.

## 데이터 흐름

초기 부트스트랩 단계의 흐름은 단순하다.

1. `uvicorn`이 `app.main:app`을 실행한다.
2. `main.py`가 설정을 로드하고 `FastAPI` 앱을 생성한다.
3. 앱이 루트 API 라우터를 등록한다.
4. 클라이언트가 `/health`를 호출하면 `health.py`가 상태 정보를 JSON으로 반환한다.

현재 단계에서는 저장소, 큐, 외부 API 호출, WebSocket 처리 흐름은 포함하지 않는다.

## 오류 처리

- 존재하지 않는 경로는 `FastAPI` 기본 404 응답에 맡긴다.
- 설정 누락이나 앱 초기화 오류는 서버 시작 실패로 빠르게 드러나도록 한다.
- 사용자 정의 예외 핸들러, 거래소 오류 매핑, 재시도 로직은 이번 범위에 포함하지 않는다.

## 검증 기준

아래 조건을 만족하면 이번 작업은 완료로 본다.

- 백엔드 의존성 설치가 완료된다.
- 개발 서버가 로컬에서 실행된다.
- `GET /health` 호출 시 200 응답과 JSON 본문이 반환된다.
- 테스트 명령이 통과한다.

## 구현 시 유의사항

- 지금 단계에서는 구조만 만들고 기능을 미리 추상화하지 않는다.
- 실시간 수집기, 전략 엔진, 주문 실행기는 이후 작업에서 별도 디렉터리로 추가한다.
- 프런트엔드와의 연동을 염두에 두되, CORS나 인증을 선행 도입하지 않는다.
- 변경 범위는 `backend/`와 관련 문서에 한정한다.

## 가정

- Python 기반 실시간 데이터 처리와 전략 계산 확장을 고려해 `FastAPI`를 선택한다.
- 현재 단계의 목적은 "운영형 백엔드"가 아니라 "실행 가능한 최소 골격"이다.
- 패키지 관리는 `pyproject.toml` 기준으로 진행한다.
